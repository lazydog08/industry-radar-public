# 一周模拟试运行记录

本记录用于验证系统在连续运行时是否符合“新增、去重、沉淀、搜索、报告归档”的目标。

## 模拟方式

- 模拟日期：`2026-05-19` 到 `2026-05-23`
- 模拟次数：5 天 × 中午/晚上 = 10 次
- 数据源：`data/sample/mock-source-items.json`
- 数据库：`data/simulations/week-2026-05-19_to_2026-05-23/industry-radar.sqlite`
- 报告目录：`data/simulations/week-2026-05-19_to_2026-05-23/reports`

可复现命令：

```bash
scripts/simulate-week.sh 2026-05-19 5 data/simulations/week-2026-05-19_to_2026-05-23
```

## 首轮发现

1. 第二天之后，中午报告仍然把历史已见过、当天再次抓到的同标题事件当作新增。
   - 影响：长期运行后每天报告会重复旧事件，不符合“快速看新增”。
   - 修复：报告分类改为只有 `first_seen_at` 落在本次窗口内才算新增；历史事件再次出现进入“持续发酵/有更新”。

2. 同一事件在同一份报告里同时出现在 Top 5、行业分类、品牌动态。
   - 影响：Markdown 复制到微信/飞书/Notion 时偏长，3 分钟阅读效率下降。
   - 修复：Top 5 已展示事件不再在后续分类块重复出现。

3. 中文多关键词搜索 `B站 流量 规则` 没有命中。
   - 影响：真实使用时常会用空格分隔多个中文关键词，单靠 FTS5 对中英文混合词切分不够稳。
   - 修复：保留 SQLite FTS5，同时增加多关键词逐词兜底搜索，标题、摘要、为什么重要、选题角度、标签、实体都会参与匹配。

4. 并发搜索同一 SQLite 数据库时，`search_history` 写入可能触发 `database is locked`。
   - 影响：Web UI 多标签页或命令行并发搜索时，搜索本身可能被非核心的历史记录写入拖垮。
   - 修复：启用 SQLite `busy_timeout`，并让搜索历史写入失败不影响实际搜索结果。

## 修复后预期

- 第一天中午：4 条新增。
- 第一天晚上：4 条新增，1 条持续更新。
- 第二天起，如果 mock 数据完全重复：中午 0 条新增，4 条持续更新；晚上 0 条新增，5 条持续更新。
- 真实运行中，只要有新标题或高相似但首次出现的新事件，会进入新增；旧事件新增来源或再次出现会进入持续更新。
