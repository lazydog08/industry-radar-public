# Assumptions

## Assumption 001
- 问题：真实 API、登录态、Cookie 或第三方平台访问可能不可用。
- 我的假设：基础可运行版优先使用公开来源；真实采集没有结果时用本地 Mock 数据兜底。
- 为什么这么假设：README 和项目设计明确要求不依赖登录、不保存账号密码，且已有 `--mock` / `--mock-fallback` 流程。
- 影响范围：报告中的部分事件可能是示例数据，不代表真实当天情报。
- 后续如何替换成真实方案：在保持合规边界的前提下继续扩展 RSS、公开接口和手工导入数据源。

## Assumption 002
- 问题：项目目标很大，但当前交付优先级是无人值守完成可演示版本。
- 我的假设：MVP 验收口径是本地可启动、可生成报告、可搜索知识库、可打开 Web UI，而不是完整覆盖所有行业和平台。
- 为什么这么假设：用户明确要求最高优先级不是完美，而是主流程能跑通。
- 影响范围：复杂语义搜索、更多数据源、长期部署细节可暂时作为后续迭代。
- 后续如何替换成真实方案：在 MVP 稳定后逐步加入 embedding、周/月深度复盘和更多来源适配器。

## Assumption 003
- 问题：当前目录不是 git 仓库。
- 我的假设：不依赖 git 来回滚或比较变更，改用 `progress.md`、验证命令和文件清单记录推进状态。
- 为什么这么假设：`git status --short` 返回不是 git 仓库。
- 影响范围：最终交付不会包含 commit 或 diff 摘要，只能列出修改过的文件。
- 后续如何替换成真实方案：如果用户之后把目录纳入 git，再用常规分支、提交和 PR 流程管理。

## Assumption 004
- 问题：Web UI 的前端交互脚本在源码目录缺失，但 `dist/web/app.js` 里存在旧构建产物。
- 我的假设：可以基于现有 `src/web/index.html`、`styles.css` 和 `dist/web/app.js` 的意图补回源码脚本，并补齐当前页面新增控件的逻辑。
- 为什么这么假设：这是最快恢复可演示主流程的路径，且不引入外部依赖。
- 影响范围：前端是轻量原生 JS，不引入 React/Vite 等额外复杂度。
- 后续如何替换成真实方案：后续如需要更复杂交互，可迁移到正式前端构建体系。

## Assumption 005
- 问题：用户问“全部跑通了吗”时，真实公开平台可能因为登录、Cookie、验证码或风控限制无法全量返回。
- 我的假设：“全部跑通”的当前验收口径是基础可运行/可演示主流程端到端通过；外部平台访问失败只要有明确降级和记录，不视为本地版本失败。
- 为什么这么假设：项目目标明确要求不要因为 API Key、登录态、真实数据或第三方服务不可用而停止，优先交付能启动、能演示、主流程能跑通的版本。
- 影响范围：演示版可以稳定展示 Mock/降级数据；真实源覆盖率仍需后续继续增强。
- 后续如何替换成真实方案：增加更多稳定 RSS、官方源和可导入数据源，并在 UI 中明确区分真实采集和 Mock/降级数据。

## Assumption 006
- 问题：当前 demo 数据使用示例 URL，但来源名仍模拟真实平台。
- 我的假设：任何带 `mock/sample/example` URL 痕迹的数据都应按示例数据处理，不能进入今日必看高分区。
- 为什么这么假设：用户明确指出信息滞后和误推风险，演示数据必须避免伪装成真实当天情报。
- 影响范围：本地 demo 首屏可能出现“今日必看 0 条”，但会诚实展示“适合做视频”和封顶原因。
- 后续如何替换成真实方案：接入真实公开源并保留真实发布时间后，Radar Score 会按真实来源、真实热度和真实时间重新排序。

## Assumption 007
- 问题：个人知识库体检需要判断哪些内容最值得补证据，但当前还没有真实二次检索任务队列。
- 我的假设：先用“封顶/示例”或“低置信 + 单来源”作为补证据优先级，用“视频潜力高且未标记已用于视频”作为可回收选题。
- 为什么这么假设：这比把所有单来源事件都列入队列更少噪音，也更贴近视频创作者的实际操作：先补可疑证据，再回收高潜力选题。
- 影响范围：演示数据因为全部是 Mock/示例会集中进入补来源队列；真实数据接入后队列会更能区分优先级。
- 后续如何替换成真实方案：把补证据队列接入真实公开源二次采集、人工导入或浏览器验证流程，并根据新增来源自动刷新置信度。
